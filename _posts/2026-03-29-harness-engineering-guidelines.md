## 에이전트 운영 규약 설계란

소프트웨어 공학의 패러다임이 AI 에이전트를 활용한 자율적 소프트웨어 인도(Software Delivery)로 전환되면서, 에이전트가 운영 환경에서 신뢰할 수 있는 방식으로 작업을 수행할 수 있도록 지원하는 **운영 규약 체계**를 설계하는 일이 중요해지고 있다. 단순히 AI 모델을 사용하는 것을 넘어, 에이전트를 둔러쌀 **시스템, 제약 조건, 피드백 루프**를 설계하는 영역이다.

핵심은 에이전트 자체가 아니라 에이전트가 활동하는 **디지털 서식지**를 구축하는 것이다. 원시 모델은 세션 간 메모리가 없고, 도구 접근에 경계가 없으며, 확신에 찬 오류를 범하는 경향이 있다. 에이전트 운영 규약 설계는 이러한 결함을 보완하고 강점을 극대화하기 위해 상태(State), 도구 실행 환경, 검증 루프, 그리고 집행 가능한 제약 조건을 제공한다.

> 에이전트가 저장소나 연결된 시스템에서 발견할 수 없는 정보는 실질적으로 존재하지 않는 것과 같다. 따라서 모든 품질 표준, 아키텍처 제약, 운영 규칙은 기계가 읽을 수 있는(Machine-readable) 형태로 명시되어야 한다.

## 4대 기둥

이 운영 규약 체계는 4개의 기둥으로 구성된다.

| 기둥 | 정의 | 서버 구현 |
|------|------|----------|
| **제약 조건과 가드레일** | 에이전트의 도구/파일 접근 제한, 위험 명령 차단 | 인프라 정책, 금지 사항 체크리스트 |
| **피드백 루프** | 작업 후 자동 검증, 실패 시 자가 수정 사이클 | ATH 추적, 코드 리뷰, 검증 절차 |
| **맥락 및 지식 주입** | 파일 시스템 기반 영구 상태, 계층적 문서 구조 | 문서 우선순위 체계, 참조 문서 |
| **관측성 및 HITL** | 행동 기록, 고위험 작업에 인간 승인 | ATH 모니터링, 승인 프로세스 |

## 우리 환경: 멀티 에이전트 홈랩

여러 AI 에이전트가 동시에 하나의 인프라에서 작업한다.

| 에이전트 | 플랫폼 | 위치 | 모델 | 역할 |
|---------|--------|------|------|------|
| 2호기 | OpenClaw | 247 Mac | Qwen3.5-397B (업그레이드 중) | 코드 작성, Gmail 연동, 외부 API, 분석, 인프라 보조 |
| 3호기 | Claude Code | 156 서버 | Claude Opus 4.6 | 인프라 관리, K8s 배포, DB 스키마, 코드 리뷰, 파이프라인 |
| 4호기 | OpenClaw | 247 Mac | ChatGPT | 설계, 분석, DDL 작성, 문서 리뷰 |

이 환경에서 운영 규약의 4대 기둥을 어떻게 적용했는지 실전 사례와 함께 정리한다.

## 기둥 1: 제약 조건과 가드레일

에이전트의 자율성이 책임(Liability)으로 변질되는 것을 방지하는 불변의 규칙이다. 우리 환경에서는 19개의 가드레일을 정의했다.

### 데이터베이스 가드레일

```
GR-DB-01: 모든 데이터 저장은 K8s PostgreSQL에 스키마 생성하여 사용
GR-DB-02: standalone Docker DB 생성 금지
GR-DB-03: DB 비밀번호 스크립트 하드코딩 금지
GR-DB-04: 새 DB/테이블 생성 전 기존 스키마 확인 필수
```

### 배포 가드레일

```
GR-K8S-01: 모든 서비스는 K8s Deployment + NodePort로 배포
GR-K8S-02: 빌드 전 gitleaks 보안 스캔 필수
GR-K8S-03: kubectl apply 후 반드시 rollout status 확인
```

### 배치 작업 가드레일

```
GR-BATCH-01: 모든 배치 작업은 Kubeflow Pipeline으로 관리
GR-BATCH-02: crontab 직접 등록 절대 금지
```

### 보안 가드레일

```
GR-SEC-01: 0.0.0.0 바인딩으로 전체 네트워크 노출 금지
GR-SEC-02: 보안 관련 작업 전 보안 정책 문서 참조 필수
```

### 왜 가드레일이 필요한가 -- 실제 사례

2호기(Qwen3.5-122B)에게 결제 메일 파싱 파이프라인 구축을 지시했을 때, 다음과 같은 일이 벌어졌다.

**사례: standalone Docker PostgreSQL 반복 생성**

K8s PostgreSQL에 이미 `finance_mail` 스키마가 구축되어 있었지만, 2호기가 접근에 실패하자 **독단적으로** Docker 컨테이너를 생성했다.

```bash
# 2호기가 생성한 컨테이너 (GR-DB-02 위반)
docker run -d -p 5432:5432 -e POSTGRES_PASSWORD=***** postgres
```

문제점:
- 비밀번호 `payments123`이 하드코딩 (GR-DB-03 위반)
- `0.0.0.0:5432`로 전체 네트워크에 노출 (GR-SEC-01 위반)
- K8s 스키마를 완전히 무시하고 별도 DB 생성 (GR-DB-04 위반)

3호기가 삭제했지만, 2호기가 **하이픈 없는 이름으로** 다시 생성했다. 총 3회 삭제해야 했다. 이 경험으로 가드레일을 명문화하고, 모든 에이전트가 작업 전 체크리스트를 확인하도록 의무화했다.

## 기둥 2: 피드백 루프

에이전트가 작업 후 자동으로 검증하고, 실패 시 자가 수정하는 사이클이다.

### 작업 수행 사이클

```
요청 접수 → ATH 등록 → 사전 분석 → 실행 → 검증 → 문서화 → ATH 완료
     ↑                                    ↓
     └──────── 검증 실패 시 되돌아감 ←──────┘
```

모든 실행 작업은 **Agent Task Hub(ATH)**에 등록해야 한다. ATH는 에이전트 간 작업 중복을 방지하고, 진행 상황을 투명하게 추적하는 오케스트레이션 허브이다.

### 코드 리뷰 분리 원칙

```
에이전트 A (작성) → 에이전트 B (리뷰) → 리뷰 통과 → 배포
```

**작성과 리뷰는 반드시 다른 에이전트가 수행한다.** 같은 에이전트가 self-approve 하는 것을 금지한다. 우리 환경에서는:
- 2호기/4호기가 코드 작성 → 3호기가 리뷰
- 4호기가 설계 제안 → 3호기가 검증 후 적용
- Antigravity가 독립 작업 수행 → 결과물을 3호기가 통합

### 실제 사례: 피드백 루프가 발견한 문제

2호기가 결제 메일 파싱 파이프라인을 구축했을 때, 3호기의 리뷰에서 다음 문제가 발견되었다.

```
301건 적재, 금액 추출 성공 22건(7.3%), 나머지 93%가 0원
```

원인: 메일 subject에서만 금액을 추출하고, body(본문)를 전혀 읽지 않았다. 추가로:
- 비결제 메일(뉴스레터, 투자메일) 혼입
- PG사(결제대행사)를 가맹점으로 분류
- 현대캐피탈=50만원, 삼성케어=1.09만원 고정 금액 하드코딩

피드백 루프가 없었다면 이 데이터가 그대로 운영에 투입되었을 것이다.

## 기둥 3: 맥락 및 지식 주입

모델의 제한된 컨텍스트 윈도우를 보완하기 위한 계층적 문서 구조이다.

### 문서 우선순위 (충돌 시 상위 우선)

```
1. 사용자 직접 지시
2. CLAUDE.md (글로벌 운영 지침)
3. agent-governance-guidelines.md (본 가이드라인)
4. 프로젝트별 CLAUDE.md
5. 에이전트별 지침 (SOUL.md 등)
```

### 디렉토리 구조

에이전트가 인식해야 할 표준 구조:

```
~/services/     → K8s 배포 서비스 코드 (Dockerfile + k8s/ 포함)
~/projects/     → 프로젝트 코드
~/scripts/      → 범용 셸 스크립트
~/documents/    → 정책 문서, 가이드
~/md/           → 문서 심볼릭 링크 허브 (RAG 최적화)
```

### 컨텍스트 윈도우 관리

| 모델 | 컨텍스트 | 관리 방법 |
|------|---------|----------|
| Claude Opus 4.6 (3호기) | 1M tokens | 자동 압축 |
| Qwen3.5-122B (2호기) | 262K tokens | auto-compaction 설정 (70%에서 트리거) |
| ChatGPT (4호기) | 128K tokens | 세션 분리 |

2호기의 경우 262K 컨텍스트가 가득 차면 응답이 불가능해진다. 실제로 742개 메시지가 누적되어 230K 토큰을 초과한 후 "inbound worker timed out" 장애가 발생했다. 이를 방지하기 위해 OpenClaw의 compaction 설정을 적용했다.

```json
{
  "compaction": {
    "reserveTokens": 65000,
    "keepRecentTokens": 50000,
    "timeoutSeconds": 1800
  }
}
```

## 기둥 4: 관측성 및 인간의 개입 (HITL)

에이전트의 모든 행동을 추적하고, 고위험 작업에 인간 승인을 요구하는 레이어이다.

### ATH 기반 작업 추적

모든 실행 작업은 ATH(Agent Task Hub)에 등록한다.

```bash
# Task 생성
ath task create --title '작업 제목' --agent claude --priority high --category development

# Task 완료
ath task update <TASK_ID> --status completed --summary '결과 요약'
```

### 인간 승인 필요 작업 분류

| 위험 등급 | 작업 유형 | 절차 |
|----------|----------|------|
| **높음** | 인프라 변경, DB 스키마 삭제, 보안 설정 | 사용자 사전 승인 필수 |
| **중간** | 새 서비스 배포, Docker 빌드 | ATH 등록 + 보안 스캔 |
| **낮음** | 코드 수정, 문서 업데이트 | ATH 등록만으로 충분 |

### 에이전트 권한 매트릭스

| 작업 | 2호기 | 3호기 | 4호기 | Antigravity |
|------|-------|-------|-------|-------------|
| K8s 배포 (kubectl) | X | **O** | X | X |
| DB 스키마 변경 | X | **O** | X (DDL 작성) | X |
| Docker 빌드 | X | **O** | X (Dockerfile 작성) | X |
| Gmail/메일 조회 | **O** | X | X | X |
| 코드 작성/분석 | **O** | O | **O** | **O** |
| 코드 리뷰 | O | **O** | **O** | O |
| SSH to DGX (228/237) | X | **O** | X | X |
| 독립 워크플로 실행 | X | X | X | **O** |

4호기가 DGX 대시보드 코드를 수정하고 이미지 빌드까지 완료했지만, `kubectl apply`에서 권한이 없어 배포가 막혔던 사례가 있다. 이는 권한 매트릭스가 정확히 작동한 예시이다.

## 에이전트 간 통신 프로토콜

```
┌─────────┐     Discord      ┌─────────┐     Discord      ┌─────────┐
│  2호기   │ ──────────────→ │  3호기   │ ←────────────── │  4호기   │
│ (247Mac) │     SSH pipe     │ (156서버) │     Discord     │ (247Mac) │
│ Gmail/코드│ ─────────────→ │ K8s/DB  │ ──────────────→ │ 설계/분석│
└─────────┘   JSON stdout    └─────────┘   지시/결과      └─────────┘
                                  ↑
                           ┌─────────────┐
                           │ Antigravity  │
                           │ (156 서버)    │
                           │ 독립 워크플로  │
                           └─────────────┘
```

- **2호기 → 3호기**: SSH 파이프로 JSON 전달, Discord 메시지
- **4호기 → 3호기**: Discord 메시지로 코드/DDL 전달, 3호기가 검증 후 적용
- **3호기 → 2호기/4호기**: Discord 메시지로 지시/결과 전달
- **Antigravity**: 독립적으로 156 서버에서 작업, 결과물을 3호기가 통합

## 금지 사항 체크리스트

모든 에이전트는 작업 완료 전 이 체크리스트를 확인한다.

- [ ] standalone Docker DB 컨테이너를 생성하지 않았는가?
- [ ] crontab/systemd timer에 등록하지 않았는가?
- [ ] DB 비밀번호를 스크립트에 하드코딩하지 않았는가?
- [ ] 0.0.0.0으로 서비스를 바인딩하지 않았는가?
- [ ] 기존 인프라/스키마를 확인하지 않고 새로 생성하지 않았는가?
- [ ] 테스트 데이터 3건 이하로 "성공"을 보고하지 않았는가?
- [ ] gitleaks 보안 스캔을 통과했는가?
- [ ] kubectl rollout status로 배포 완료를 확인했는가?

## 결론

에이전트 운영 규약 설계는 AI 에이전트에게 **자유를 주되, 울타리 안에서만** 작동하게 하는 체계다. 4개의 AI 에이전트가 동시에 하나의 인프라에서 작업할 때, 명문화된 가드레일 없이는 같은 실수가 반복된다.

가장 중요한 교훈:
1. **가드레일은 사후 대응이 아니라 사전 설계**여야 한다
2. **피드백 루프 없이 배포하면 93%가 0원인 데이터가 운영에 올라간다**
3. **에이전트가 읽을 수 없는 규칙은 존재하지 않는 것과 같다**
4. **작성과 리뷰는 반드시 분리**해야 한다

*"에이전트가 저장소나 연결된 시스템에서 발견할 수 없는 정보는 실질적으로 존재하지 않는 것과 같다."*
